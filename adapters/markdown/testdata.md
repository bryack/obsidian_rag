```go
package sync

import (
	"sync"
	"testing"
)

func TestCounter(t *testing.T) {
	t.Run("incrementing the counter 3 times leaves it at 3", func(t *testing.T) {
		counter := Counter{}
		counter.Inc()
		counter.Inc()
		counter.Inc()

		if counter.Value() != 3 {
			t.Errorf("got %d, want %d", counter.Value(), 3)
		}
	})

	t.Run("it runs safely concurrently", func(t *testing.T) {
		wantedCount := 1000
		counter := Counter{}

		var wg sync.WaitGroup
		wg.Add(wantedCount)

		for i := 0; i < wantedCount; i++ {
			go func() {
				counter.Inc()
				wg.Done()
			}()
		}
		wg.Wait()

		if counter.Value() != wantedCount {
			t.Errorf("want %d, but got %d", counter.Value(), wantedCount)
		}
	})
}
```

```go
package sync

import "sync"

type Counter struct {

func (c *Counter) Inc() {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.value++
}

}
```

# Sync
Мы хотим создать счётчик, который безопасно использовать в конкурентной среде.

Начнём с небезопасного счётчика и убедимся, что он работает в однопоточной среде.

Затем проверим его небезопасность: несколько горутин будут пытаться использовать счётчик в тесте, и мы это исправим.

### Пишем тест первым

Мы хотим, чтобы наш API предоставлял метод для увеличения счётчика и метод для получения его значения.

```go
func TestCounter(t *testing.T) {
	t.Run("incrementing the counter 3 times leaves it at 3", func(t *testing.T) {
		counter := Counter{}
		counter.Inc()
		counter.Inc()
		counter.Inc()

		if counter.Value() != 3 {
			t.Errorf("got %d, want %d", counter.Value(), 3)
		}
	})
}
```

```go
package sync

type Counter struct {
	value int
}

func (c *Counter) Inc() {
	c.value++
}

func (c *Counter) Value() int {
	return c.value
}
```
### Следующие шаги

Это было достаточно просто, но теперь у нас есть требование: счётчик должен быть безопасен для использования в конкурентной среде. Нам нужно написать падающий тест, чтобы проверить это.

### Пишем тест первым

```go
t.Run("it runs safely concurrently", func(t *testing.T) {
	wantedCount := 1000
	counter := Counter{}

	var wg sync.WaitGroup
	wg.Add(wantedCount)

	for i := 0; i < wantedCount; i++ {
		go func() {
			counter.Inc()
			wg.Done()
		}()
	}
	wg.Wait()

	assertCounter(t, counter, wantedCount)
})
```

Этот тест будет проходить по `wantedCount` и запускать горутину для вызова `counter.Inc()`.

Мы используем `sync.WaitGroup` — удобный способ синхронизации конкурентных процессов.

> `WaitGroup` ждёт завершения набора горутин. Основная горутина вызывает `Add`, чтобы установить количество горутин, которые нужно дождаться. Затем каждая горутина запускается и вызывает `Done` по завершении. Одновременно можно использовать `Wait`, чтобы заблокировать выполнение, пока все горутины не закончат работу.

Дожидаясь завершения `wg.Wait()` перед проверками, мы можем быть уверены, что все наши горутины попытались увеличить счётчик.

### Пробуем запустить тест

```
=== RUN   TestCounter/it_runs_safely_in_a_concurrent_envionment
--- FAIL: TestCounter (0.00s)
    --- FAIL: TestCounter/it_runs_safely_in_a_concurrent_envionment (0.00s)
    	sync_test.go:26: got 939, want 1000
FAIL
```

Тест, вероятно, упадёт с другим числом, но в любом случае он демонстрирует, что счётчик не работает, когда несколько горутин одновременно пытаются изменить его значение.

### Напишем достаточно кода, чтобы тест прошёл

Простое решение — добавить блокировку к нашему `Counter`, чтобы только одна горутина могла увеличивать счётчик в данный момент. Для этого в Go есть `Mutex`:

`Mutex` — это блокировка взаимного исключения. Нулевое значение `Mutex` — разблокированная блокировка.

```go
type Counter struct {
    mu    sync.Mutex
    value int
}
```

```go
func (c *Counter) Inc() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value++
}
```

Это означает, что любая горутина, вызывающая `Inc`, захватит блокировку на `Counter`, если она первая. Все остальные горутины будут ждать, пока блокировка не будет снята, прежде чем получат доступ.

Если теперь перезапустить тест, он должен пройти, потому что каждая горутина ждёт своей очереди перед изменением значения.

Я видел другие примеры, где `sync.Mutex` встраивается в структуру.

Можно встретить такие примеры:

```go
type Counter struct {
    sync.Mutex
    value int
}
```

Можно утверждать, что это делает код немного элегантнее.

```go
func (c *Counter) Inc() {
    c.Lock()
    defer c.Unlock()
    c.value++
}
```

Это выглядит красиво, но хотя программирование — в значительной степени субъективная дисциплина, это плохо и неправильно.

Иногда люди забывают, что встраивание типов означает, что методы этого типа становятся частью публичного интерфейса; а это часто нежелательно. Помни, что с публичными API нужно быть очень осторожными: как только мы делаем что-то публичным, другой код может привязаться к этому. Мы всегда хотим избегать ненужной привязки.

Открытие `Lock` и `Unlock` в лучшем случае сбивает с толку, а в худшем — может быть очень вредным для программы, если пользователи типа начнут вызывать эти методы.

Это кажется действительно плохой идеей.

### Копирование мьютексов

Наш тест проходит, но код всё ещё немного опасен.

Если запустить `go vet` на коде, должна появиться ошибка вроде:

```
sync/v2/sync_test.go:16: call of assertCounter copies lock value: v1.Counter contains sync.Mutex
sync/v2/sync_test.go:39: assertCounter passes lock by value: v1.Counter contains sync.Mutex
```

В документации `sync.Mutex` объясняется почему:

> `Mutex` нельзя копировать после первого использования.

Когда мы передаём наш `Counter` (по значению) в `assertCounter`, происходит попытка создать копию мьютекса.

Чтобы это исправить, нужно передавать указатель на `Counter`, поэтому изменим сигнатуру `assertCounter`:

```go
func assertCounter(t testing.TB, got *Counter, want int)
```

Наши тесты больше не компилируются, потому что мы пытаемся передать `Counter`, а не `*Counter`. Чтобы это решить, я предпочитаю создать конструктор, который покажет пользователям API, что лучше не инициализировать тип самостоятельно.

```go
func NewCounter() *Counter {
    return &Counter{}
}
```

Используй эту функцию в тестах при инициализации `Counter`.

### Подводим итоги

Мы рассмотрели несколько вещей из пакета sync:
- `Mutex` позволяет добавлять блокировки к нашим данным.
- `WaitGroup` — средство для ожидания завершения работы горутин.

Когда использовать блокировки вместо каналов и горутин?

Мы ранее разбирали горутины в первой главе о конкурентности, которые позволяют писать безопасный конкурентный код, так зачем же использовать блокировки? На вики Go есть страница, посвящённая этой теме: [Mutex Or Channel](https://go.dev/wiki/MutexOrChannel) (пересказ [[Sync#Мьютекс или канал в Go?]])

Распространённая ошибка новичков в Go — чрезмерное использование каналов и горутин только потому, что это возможно и/или потому, что это интересно. Не бойтесь использовать `sync.Mutex`, если это лучше всего подходит для вашей задачи. Go прагматичен: он позволяет использовать инструменты, которые лучше всего решают вашу проблему, и не заставляет придерживаться одного стиля кода.

Перефразируя:
- Используйте каналы, когда передаёте владение данными.
- Используйте мьютексы для управления состоянием.

### go vet

Не забывайте использовать `go vet` в ваших скриптах сборки — он может предупредить о некоторых тонких багах в коде до того, как они попадут к вашим несчастным пользователям.

### Не используйте встраивание только потому, что это удобно

Подумайте о влиянии встраивания на ваш публичный API.
Действительно ли вы хотите выставлять эти методы и позволять другим привязывать свой код к ним?

В случае с мьютексами это может быть потенциально катастрофично непредсказуемыми и странными способами: представьте, что какой-то вредоносный код разблокирует мьютекс, когда этого делать не следует — это вызовет очень странные баги, которые будет трудно отследить.

# Мьютекс или канал в Go?

Статья объясняет, как синхронизировать горутины в Go: использовать каналы или мьютексы из пакета `sync`. Главный принцип Go — "делиться памятью через коммуникацию, а не общаться через разделяемую память". Но иногда мьютексы проще и лучше.[go](https://go.dev/wiki/MutexOrChannel)​

## Когда что использовать

|Каналы|Мьютексы|
|---|---|
|Передавать данные, распределять задачи, получать результаты асинхронно|Кэши, состояние программы|

**Правило:** Выбирай то, что проще и понятнее. Новички часто переусердствуют с каналами "потому что круто". Не бойся мьютексов — Go позволяет использовать оба подхода.[go](https://go.dev/wiki/MutexOrChannel)​

## WaitGroup для ожидания

`sync.WaitGroup` помогает дождаться завершения группы горутин. Полезно для:
- Чистого завершения программы
- Циклических алгоритмов, где горутины работают параллельно, ждут барьер, потом снова.[go](https://go.dev/wiki/MutexOrChannel)​
    
## Итог для новичка

Каналы, мьютексы и WaitGroup дополняют друг друга. Используй:
- **Каналы** — когда горутины общаются
- **Мьютекс** — когда защищаешь общую переменную
- **WaitGroup** — когда ждёшь все горутины

Если мьютексы усложняют код — переделай на каналы.[go](https://go.dev/wiki/MutexOrChannel)​
