# Obsidian RAG: Персональный ИИ-ассистент для глубокого анализа знаний


Obsidian RAG — это высокопроизводительная локальная система поиска и генерации ответов (Retrieval-Augmented Generation), разработанная специально для работы с объемными и разрозненными базами заметок в Obsidian.


## Мотивация проекта
При работе с большими базами знаний (9500+ заметок) стандартные решения часто дают сбой:
1. Контекстное окно: Простое подключение LLM к папке с заметками быстро переполняет память модели, из-за чего она теряет нить разговора.
2. Потеря информации: Поиск по ключевым словам не находит данные, записанные несколько лет назад другими словами.
3. Приватность: Личные заметки не должны покидать локальную машину.

Этот проект решает данные проблемы, используя гибридный поиск и Clean Architecture для обеспечения точности и масштабируемости.


## Ключевые особенности
- Hybrid Search (Dense + Sparse): Комбинирует семантический поиск (векторы BGE-M3) с лексическим (разреженные векторы), что позволяет находить как концепции, так и точные термины.
- Scoped Search: Возможность ограничивать область поиска конкретными папками или проектами без потери производительности.
- Clean Architecture (Hexagonal): Полное разделение бизнес-логики и инфраструктуры (Qdrant, Ollama, FileSystem), что делает систему расширяемой и тестируемой.
- TDD в основе: Проект разработан по методологии Test-Driven Development с использованием интеграционных тестов на Testcontainers.


## Технологический стек
- Язык: Go (Golang) — выбран за производительность и эффективную работу с параллельными задачами.
- Векторная БД: Qdrant (https://qdrant.tech/) (gRPC) — используется для хранения эмбеддингов и метаданных.
- LLM Инференс: Ollama (https://ollama.com/) (модели bge-m3 для эмбеддингов и qwen3.5 для генерации).
- Markdown Parser: Goldmark (https://github.com/yuin/goldmark) с расширениями для обработки Wikilinks и Frontmatter.


## Архитектура системы
Проект следует принципам Hexagonal Architecture:
- internal/domain: Ядро системы, содержащее бизнес-правила (RagEngine) и интерфейсы.
- adapters/: Реализации для внешних систем (Qdrant, Ollama, Markdown-парсинг).
- cmd/cli: Интерфейс командной строки.

Такой подход позволил внедрить сложные зависимости (например, разные модели для поиска и ответов) без замусоривания основной логики.

## Инженерные вызовы и решения

1. Стабилизация GPU и «Эффект наблюдателя»
В процессе разработки была выявлена проблема: при мгновенном переключении Ollama с модели эмбеддингов на модель генерации на картах с 8GB VRAM (RTX 3070) возникали ошибки доступа к контексту.
Решение: Внедрен механизм стратегической задержки (200мс) и изоляция HTTP-клиентов, что позволило стабилизировать переключение контекста GPU и обеспечить 100% надежность ответов.


2. Оптимизация под локальное железо
Система оптимизирована для работы на потребительских видеокартах (8GB VRAM):
    - Использование квантованных моделей (qwen3.5:9b).
    - Умное управление контекстным окном (лимит 12,000 символов с обрезкой по границам слов).
    - Настройка параметров keep_alive для эффективного освобождения видеопамяти.


## Быстрый старт

### Предварительные требования
- Go 1.25+
- Docker (для Qdrant)
- Ollama с загруженными моделями bge-m3 и qwen3.5:9b


### Запуск
1. Проиндексировать Vault:
```bash
go run cmd/cli/main.go index /path/to/obsidian
```
2. Задать вопрос (режим поиска):
```bash
go run cmd/cli/main.go ask /path/to/obsidian "Как настроить Docker?"
```
3. Получить умный ответ (режим RAG):
```bash
go run cmd/cli/main.go -llm ask /path/to/obsidian "Сформулируй краткое саммари по проекту X"
```

### Тестирование
Проект покрыт тестами на разных уровнях:
- Unit-тесты: Использование Spies и Stubs для проверки логики RagEngine без вызова реальных моделей.
- Integration-тесты: Реальный запуск Qdrant в Docker-контейнерах через Testcontainers-go для проверки контрактов БД.

Запуск тестов:
```bash
go test ./...
```
